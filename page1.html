Week 4

I decided to come into project a few hours early in order to sort out the problem with the virtual machine, as it would be a bad look if we were stuck unable to access our VM during class time. After getting the login user from Roy and his copy of the private and public keys, I began attacking the problem. After getting a similar error to Roy, I decided to generate new keys using PuttyGen, which I was able to apply to the virtual machine under the 'Passwords' section of the VM's Azure page. Doing so allowed me to connect via the command line with the command, ssh -i RT-Admin@20.193.69.242, or directly through Putty. 

Once Roy arrived, we began installing the dependencies required to run RT Resource Tracker. This began with installs of MariaDB and Apache2. Initially, we'd gone with the idea that we were going to run our database on a separate VM, due to a set of requirements we'd read for RT. However, after a bit of discussion, we agreed that it really wasn't necessary at all and only served to inflate our costs needlessly. RT requires a lot of apache2 modules, something that I'd previously had no experience in working with, the process turned out to be barely different to installing any other program, simply being a slightly modified apt get command, sudo apt-get install libapache2-mod-fcgid, for example.

A major problem that we were running into was that the documentation for RT was hugely out of date, the wiki was a series of dead links and miss-information, as a result of this we spent quite a long period of time attempting to cobble together a strategy that would successfully give us an install. Executing a wget command on the RT GitHub I downloaded the directory and unpacked it onto our VM. RT comes with several commands for an easy instillation, checking dependency errors as well as automatically deploying, however, what we didn't know was the dependencies that we needed in order to be able to run these commands in the first place. This resulted in quite a lot of trial and error as we struggled to get the program to function at all.

Week 7

Returning to campus on Monday, I was working alone as Roy had injured his leg at the gym and was in a physical therapy appointment. The majority of the period was taken up by group meetings to discuss where we were at and how we were going to progress forward. I created a new Virtual Machine, titled RT20.04, my thought process being that we'd gotten a lot further in terms of dependencies and that a fresh start would be ideal for ensuring that we didn't run into messy errors down the line from our experimentation period. 

I opted to ignore the automatic dependency programs that came with RT, deciding that ultimately they were far too much trouble than they were worth. By this point I had managed to piece together exactly what was required, so installing them all manually was fairly trivial. After setting up the database on Maria and manually installing each program, I was finally given the all clear by RT for all dependencies. From where it was simply a matter of configuring Apache to point towards my RT directory and I was able to generate a complete deployment.

Logging in with default credentials of Root and Password, I was able to access the site fully, however whenever I tried to create something or edit a user, I.E interact with the database, I was given a Cross-site request forgery (CSRF) error. I believed that this was to do with the ip address for RT not being whitelisted for access, thus the system was blocking itself in order to preserve data integrity. Unsure of where to go from here, I began researching a solution.

RT stores it's configuration files for whitelisting in /opt/rt5/etc/RT_SiteConfig.pm, once I found this, it was as simple as adding RT20.04's IP address to the list. From here the CSRF error was completely resolved.

Week 14

As Roy was now fully off working on Isaacrt, I decided to dedicate my time to getting the SSL working. After digging around in the teams files I was able to find files specific for our DNS op-bit.nz, these being a ca-bundle, crt, key and pem file alongside instructions for how to go about it. In practice however, I found that it wasn't going to be as simple as advertised. Apache's ssl folder is currently considered bad practice by users, the suggestion now is to store them within a different directory, /etc/ssl/certs. From here you would need to then enable SSL within the rt5.conf file within Apache. I was getting an error where the insecure version of the website was working fine while the htttps:// prefix caused it break. After tinkering through I was able to create a dummy page on Apache using the files supplied by Rob for security. This did end up working for a secure site, however I struggled with getting it working on the original. In the end I decided to rewrite the entire rt5.conf file manually, putting in a clause where the insecure version of the site immediately redirects towards the https version. After an hour or two of bug fixing I had it ready to go.

With this out of the way, I was able to turn my attention completely towards my my Isaacrt write up. I wanted to have instillation instructions included, as this was something we were never able to find when attempting it ourselves. This meant in the rare event that the site did go down or was somehow removed they would be able to quickly build a replacement. The completed write up can be viewed Here.
